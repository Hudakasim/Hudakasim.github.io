Gez — Autonomous Differential-Drive Robot

Personal Project
2025 — 2026
Solo project

Overview

Gez is a differential-drive robot designed and built from scratch. The system runs ROS 2 Humble on a Raspberry Pi 4 for high-level navigation and SLAM, while an ESP32 handles real-time motor control with discrete PID, encoder-based odometry, and dual-core RTOS — connected via micro-ROS over USB serial. The project covers everything from low-level motor control to autonomous navigation with LiDAR.

The Gez robot with all components mounted: ESP32, Raspberry Pi 4, RPLIDAR C1, motors, and encoders.

System Architecture

The system uses a dual-computer architecture, which is standard in professional robots like TurtleBot and Husky. The ESP32 handles real-time motor control because Linux (on the Raspberry Pi) is not deterministic enough for PID loops. The Raspberry Pi handles ROS 2, SLAM, and Nav2 because the ESP32 doesn't have enough RAM and CPU for these tasks.

Mechanical Design

The robot uses a differential-drive configuration with two independently driven wheels (65mm diameter, 169mm apart) and a caster wheel at the rear. The chassis consists of two stacked plates with standoffs for component mounting.

Low-Level Control (ESP32)

The ESP32 runs the complete real-time control stack using its dual-core architecture: Core 0 handles micro-ROS communication, while Core 1 runs the PID control loop at 10 Hz (100ms period). Each PID cycle reads encoders using the hardware PCNT module (no CPU overhead), applies a low-pass filter, ramps the target velocity for smooth acceleration, computes discrete PID, and writes PWM to the motors.

The PID gains (Kp=150, Ki=600, Kd=1) were first designed in MATLAB using the ITAE method on a motor transfer function model, then fine-tuned on the real robot. Anti-windup is implemented by clamping the integral term to ±5, preventing overshoot when the motor reaches PWM saturation.

Kinematics & Odometry

The robot implements both inverse and forward kinematics for differential drive. Inverse kinematics converts ROS 2 velocity commands (linear.x, angular.z) to individual wheel speeds. Forward kinematics (odometry) integrates encoder readings using Euler integration to estimate the robot's position (x, y, theta) in the world frame. The odometry is published at 20 Hz on the /odom topic.

micro-ROS Communication

The ESP32 acts as a micro-ROS client, communicating with the ROS 2 network through a micro-ROS agent on the Raspberry Pi via USB serial at 921,600 baud. The ESP32 subscribes to /cmd_vel and publishes /odom and /joint_states. A state machine manages the connection lifecycle: WAITING → CONNECTING → CONNECTED → DISCONNECTED, with automatic motor stop on disconnect as a safety measure.

Gez robot visualized in RViz with the full TF tree and joint states.

ROS 2 Integration

The robot is fully described in URDF with all links (base_link, wheels, caster, LiDAR mount) and joints. The robot_state_publisher reads the URDF and joint states to broadcast the complete TF tree. The TF hierarchy is: map → odom (from SLAM) → base_link (from ESP32 odometry) → wheel/lidar frames (from URDF).

SLAM & Navigation

RPLIDAR C1 is integrated and provides 360° laser scans for SLAM Toolbox mapping. The robot builds a 2D occupancy grid map while being driven manually, then uses Nav2 for autonomous navigation to goal poses within the mapped environment.

Key Design Decisions

  • Dual-computer architecture: ESP32 for deterministic real-time control, RPi for compute-heavy ROS 2 tasks
  • Hardware PCNT for encoder counting: zero CPU overhead for pulse counting
  • Pinned PID task to Core 1: isolates time-critical control from network I/O on Core 0
  • Low-pass filter on encoder readings: removes noise while preserving responsiveness (alpha=0.3)
  • Velocity ramp: limits acceleration to 1.5 rad/s per cycle, preventing PID overshoot from sudden commands
  • Safety state machine: automatic motor stop when micro-ROS connection drops
  • Motor calibration: different PWM start thresholds (100 vs 300) and max speeds (0.39 vs 0.45 m/s) between left and right motors, compensated in software
